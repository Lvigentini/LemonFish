# OASIS Engine — Developer Notes

Secondary reference doc capturing what we know about the upstream **camel-oasis** simulation engine that LemonFish depends on. This is not a user guide — it's for developers considering modifications to the simulation layer.

**Upstream:** [camel-ai/oasis](https://github.com/camel-ai/oasis) · `camel-oasis==0.2.5` · pinned via camel-ai 0.2.78

**Audit date:** 2026-04-12 against the installed package inside the LemonFish slim Docker image (`/app/backend/.venv/lib/python3.11/site-packages/oasis/`).

## Part 1 — Agent persona structure

LemonFish generates personas per entity and feeds them to OASIS. This section documents what's actually in a persona and which fields influence behaviour.

### 1.1 Field inventory

The `OasisAgentProfile` dataclass (`backend/app/services/oasis_profile_generator.py:45-156`) has 17 fields:

| Field | Type | Source | Stability |
|-------|------|--------|-----------|
| `user_id` | `int` | Sequential assignment | Stable |
| `user_name` | `str` | `entity.name` + 3-digit suffix | Randomised suffix |
| `name` | `str` | Zep entity | Stable |
| `bio` | `str` | LLM (200 chars) | Non-det (T=0.7) |
| `persona` | `str` | LLM (2000 chars) | Non-det (T=0.7) |
| `karma` | `int` | `random.randint(500, 5000)` | Random |
| `friend_count` | `int` | `random.randint(50, 500)` | Random |
| `follower_count` | `int` | `random.randint(100, 1000)` | Random |
| `statuses_count` | `int` | `random.randint(100, 2000)` | Random |
| `age` | `int?` | LLM | Non-det |
| `gender` | `str?` | LLM | Non-det (but constrained to `male`/`female`/`other`) |
| `mbti` | `str?` | LLM | Non-det (16 types) |
| `country` | `str?` | LLM or random fallback | Non-det |
| `profession` | `str?` | LLM or entity attribute | Semi-det |
| `interested_topics` | `List[str]` | LLM | Non-det |
| `source_entity_uuid` | `str?` | Zep UUID | Stable |
| `source_entity_type` | `str?` | Zep label | Stable |
| `created_at` | `str` | `datetime.now()` | Stable |

### 1.2 Which fields actually reach the agent's system prompt

This is the most important finding: **most of the persona is cosmetic**. Only a handful of fields are injected into the OASIS agent's system prompt and therefore influence behaviour.

**Important correction (2026-04-12 audit):** the MBTI / age / gender / country fields are **only used by the Reddit agent system prompt, not Twitter**. The exact line is in `oasis/social_platform/config/user.py:93-100`:

```python
# Only appears in to_reddit_system_message() — NOT to_twitter_system_message()
description += (
    f"You are a {self.profile['other_info']['gender']}, "
    f"{self.profile['other_info']['age']} years old, with an MBTI "
    f"personality type of {self.profile['other_info']['mbti']} from "
    f"{self.profile['other_info']['country']}."
)
```

The Twitter system prompt only uses the `user_profile` string — it doesn't touch `mbti`, `gender`, `age`, or `country` at all.

| Tier | Fields | Impact |
|------|--------|--------|
| **Critical** | `persona` (2000-char description) | The only descriptive text injected into both Reddit and Twitter agent system prompts. Everything the agent "is" comes from this string. |
| **Reddit-only** | `age`, `gender`, `mbti`, `country` | One sentence appended to the Reddit agent system prompt ("You are a male, 42 years old, with an MBTI personality type of INTJ from Australia."). Twitter discards these fields entirely. |
| **Low / orphaned** | `profession`, `interested_topics` | Generated by LLM, shown in UI, **never reach the agent system prompt on either platform**. |
| **Window dressing** | `bio`, `karma`, `friend_count`, `follower_count`, `statuses_count`, `user_name` | Display-only. Agent behaviour ignores them. |

Source: the exact fields that flow into SocialAgent are visible in `backend/scripts/oasis_model_patch.py:102-136` (reddit) and `:147-202` (twitter). Everything else is stripped before the agent is constructed.

### 1.2a Why MBTI was a poor choice — and what replaced it

After discovering that MBTI is just a one-sentence mention in the Reddit-only system prompt, we audited the rationale for using it at all and found:

1. **No written rationale** — neither OASIS nor LemonFish documented why MBTI was chosen over Big Five, HEXACO, or any other framework.
2. **MBTI is considered pseudoscientific** in academic personality psychology — it has poor test-retest reliability, low predictive validity, and its type categories aren't supported by factor analysis.
3. **The "institutions are all ISTJ" lock** was a clear admission that MBTI doesn't map to organisations at all.
4. **The `persona` string does almost all the real work** — the MBTI label mostly adds noise to an already-rich narrative.

**Decision (Phase 7.16):** replace MBTI with research-backed frameworks:
- **Big Five (FFM) for individuals** — 5 dimensions scored 0-100, direct empirical backing
- **Organisational archetypes + behavioural traits for institutions** — 8 archetypes inspired by Mintzberg/Miles&Snow plus 5 behavioural dimensions (formality, risk tolerance, transparency, responsiveness, ideological intensity)
- **MBTI kept as a legacy display field** — not used for behaviour, not used in LLM prompts

Implementation lives in `backend/app/services/personality_frameworks.py`. Full rationale, citations, and taxonomies are in [`docs/personality_frameworks.md`](./personality_frameworks.md).

### 1.2b How the new frameworks reach the agent

The new fields reach the agent via the **same path as before** — the `persona` string. During profile generation, we:

1. Ask the LLM to produce Big Five scores (individuals) or archetype + org_traits (institutions), grounded in the Zep entity's source material
2. Store the structured fields as dedicated columns for UI display and analytics
3. **Also render the structured fields as a natural-language paragraph appended to the persona string** (`personality_frameworks.big_five_narrative()` / `org_traits_narrative()`)

That appended paragraph flows through to OASIS's existing `user_profile` injection in `UserInfo.to_reddit_system_message()` / `to_twitter_system_message()` — no upstream OASIS changes required. The agent sees something like:

> "Your have profile: [2000 chars of persona]. Personality profile (Big Five): high openness (82/100) — curious, inventive, drawn to novelty and abstract ideas; low agreeableness (30/100) — skeptical, competitive, willing to challenge and argue; ..."

or for an institution:

> "Your have profile: [2000 chars of persona]. Organisational archetype: Advocacy. NGOs, campaigns, pressure groups, unions. Passionate, confrontational, morally framed messaging. Behavioural profile: low formality (25/100) — casual, colloquial; high risk_tolerance (85/100) — willing to engage controversy head-on; ..."

This means **both Twitter and Reddit agents now receive personality information** (previously only Reddit did via the MBTI line).

### 1.3 Individual vs Group persona pathways

Two code paths based on entity type (`oasis_profile_generator.py:186-195`):

```python
INDIVIDUAL_ENTITY_TYPES = [
    "student", "alumni", "professor", "person", "publicfigure",
    "expert", "faculty", "official", "journalist", "activist"
]

GROUP_ENTITY_TYPES = [
    "university", "governmentagency", "organization", "ngo",
    "mediaoutlet", "company", "institution", "group", "community"
]
```

Key differences:

| Attribute | Individual | Group / Institution |
|-----------|-----------|---------------------|
| `age` | LLM-inferred | **Fixed 30** (virtual) |
| `gender` | `male` / `female` | **Fixed `other`** |
| `mbti` | any of 16 types | **Fixed `ISTJ`** |
| Persona prompt focus | personal behaviour, emotional reactions, lived experience | official stance, brand voice, stakeholder engagement, controversy handling |
| System prompt tone | "share perspectives, connect with peers" | "communicate official positions, engage stakeholders" |

**Weakness:** locking every institution to `ISTJ + age 30 + gender other` flattens their differentiation. A scrappy NGO and a government ministry get identical metadata; all differentiation must come from the narrative `persona` text.

### 1.4 LLM prompts — key excerpts

System prompt (both individual and group) — `oasis_profile_generator.py:693-794`:

```
你是社交媒体用户画像生成专家。生成详细、真实的人设用于舆论模拟,
最大程度还原已有现实情况。必须返回有效的JSON格式，所有字符串值
不能包含未转义的换行符。
```

Individual persona fields requested:

1. Basic info — age, profession, education, location
2. Background — key events, relationship to the simulation topic, social connections
3. Personality — MBTI, core traits, emotional expression style
4. Social media behaviour — posting frequency, content preferences, interaction style, language patterns
5. Stances — attitudes toward the topic, what triggers positive/negative reactions
6. Unique traits — catchphrases, special experiences, hobbies
7. Personal memories — specific connection to the event, prior actions and reactions

Group persona fields requested:

1. Institutional info — formal name, type, founding, mandate
2. Account positioning — type, target audience, core function
3. Communication style — language, common expressions, taboo topics
4. Content patterns — types, frequency, active hours
5. Stances — official position on core topics, controversy handling
6. Identity — represented group, operational habits
7. Institutional memory — event connection, prior actions and reactions

### 1.5 Reproducibility

Personas are generated at `temperature=0.7` with **no seed**. Regenerating profiles for the same entities produces different personas each time.

**Implication for research use:** experiments are not reproducible unless you preserve the generated `reddit_profiles.json` / `twitter_profiles.csv` across runs. Same-seed reproducibility would need:
1. Expose `temperature` as a configurable per-step (trivial — Phase 2 already supports step-specific config)
2. Pass an explicit `seed` parameter to OpenAI calls (not all providers honour this)
3. Cache generated profiles by content hash to avoid regeneration

### 1.6 Serialisation — Reddit vs Twitter

Reddit uses JSON (`reddit_profiles.json`):

```json
[
  {
    "user_id": 0,
    "username": "entity_name_123",
    "name": "Entity Name",
    "bio": "200-char summary",
    "persona": "2000-char detailed persona",
    "karma": 1000,
    "created_at": "2026-04-10",
    "age": 30,
    "gender": "male",
    "mbti": "INTJ",
    "country": "United States",
    "profession": "professor",
    "interested_topics": ["policy", "education"]
  }
]
```

Twitter uses CSV (`twitter_profiles.csv`) with only 5 columns:

```csv
user_id,name,username,user_char,description
0,"Entity Name","entity_name_123","bio + persona combined (newlines stripped)","bio only"
```

Twitter compresses everything into `user_char`. Reddit preserves the structured fields. This is an upstream OASIS choice, not ours.

## Part 2 — Platform support in OASIS

### 2.1 What's supported

```python
# oasis/social_platform/typing.py
class DefaultPlatformType(Enum):
    TWITTER = "twitter"
    REDDIT = "reddit"
```

**Two platforms. Full stop.** No LinkedIn, Instagram, TikTok, Bluesky, Threads, or anything else.

### 2.2 Why these two

No explicit rationale in comments or docs, but the architecture makes the research intent clear: these represent **opposing paradigms** of social media.

| Dimension | Twitter | Reddit |
|-----------|---------|--------|
| **Feed algorithm** | Personalised ML (TWHIN-BERT embeddings) | Global hot score (likes/dislikes × recency) |
| **Social graph** | Follow-based, asymmetric | Community-based, egalitarian |
| **Content structure** | Flat posts + reposts/quotes | Posts with nested comment trees |
| **Actions** | 6 (post, like, repost, follow, quote, do-nothing) | 13 (add comments, dislikes, search, mute, trend) |
| **Self-rating allowed** | No | Yes |
| **Score visible** | No (likes/dislikes shown separately) | Yes (likes − dislikes) |
| **Time handling** | Integer time steps | Real datetime objects |

Twitter models the "personalised algorithmic feed". Reddit models "communal voting + threaded discussion". Between them they cover the bulk of what opinion simulation research cares about.

### 2.3 Action space per platform

All actions (`oasis/social_platform/typing.py`):

```python
class ActionType(Enum):
    EXIT, REFRESH, SEARCH_USER, SEARCH_POSTS, CREATE_POST, LIKE_POST,
    UNLIKE_POST, DISLIKE_POST, UNDO_DISLIKE_POST, REPORT_POST, FOLLOW,
    UNFOLLOW, MUTE, UNMUTE, TREND, SIGNUP, REPOST, QUOTE_POST,
    UPDATE_REC_TABLE, CREATE_COMMENT, LIKE_COMMENT, UNLIKE_COMMENT,
    DISLIKE_COMMENT, UNDO_DISLIKE_COMMENT, DO_NOTHING, PURCHASE_PRODUCT,
    INTERVIEW, JOIN_GROUP, LEAVE_GROUP, SEND_TO_GROUP, CREATE_GROUP,
    LISTEN_FROM_GROUP
```

All actions are implemented in the Platform class, but agents are restricted to a platform-specific subset at construction time:

- `get_default_twitter_actions()` → 6 actions
- `get_default_reddit_actions()` → 13 actions

Notably unused by the default MiroFish scripts: `INTERVIEW`, `PURCHASE_PRODUCT`, group-related actions (`JOIN_GROUP`, `LEAVE_GROUP`, etc). OASIS can do more than Twitter/Reddit allow — the platform defaults are a subset choice.

### 2.4 Recommendation system differences

**Twitter — TWHIN-BERT personalised:**
- Transformer embeddings on user profile + post content
- Factors: user bio, historical posts, post content vector, recency (logarithmic decay)
- Result: per-user personalised feed based on semantic relevance
- File: `oasis/social_platform/recsys.py` — `rec_sys_personalized_twh()` (lines 419-681)

**Reddit — Hot score:**
```python
score = sign(likes - dislikes) * log10(abs(likes - dislikes)) + (timestamp_seconds / 45000)
```
- Statistical ranking, no personalisation
- All users see roughly the same ranked feed
- File: `oasis/social_platform/recsys.py` — `calculate_hot_score()` (lines 168-210), `rec_sys_reddit()` (lines 213-260)

**Twitter follow-graph feed query** (`platform.py:280-313`):
```sql
SELECT post.post_id FROM post
JOIN follow ON post.user_id = follow.followee_id
WHERE follow.follower_id = ?
ORDER BY post.num_likes DESC LIMIT ?
```

### 2.5 What the missing platforms would add

| Platform | Unique mechanic | Simulation value |
|----------|----------------|------------------|
| **LinkedIn** | Professional identity, endorsements, employment graph, B2B content | **High** for business simulations (council outreach, B2B marketing, stakeholder networks) |
| **Instagram** | Visual content, ephemeral stories, influencer hierarchies | Medium — content is fundamentally visual/video, hard to simulate via text-only LLMs |
| **TikTok** | Algorithmic feed ignoring social graph, creator economy, rapid trend cycles | **High** for viral/meme dynamics but same visual-media constraint |
| **Bluesky / Threads** | Federated/ActivityPub, per-instance moderation variance | Niche |
| **Discord / private groups** | Closed communities, role-based permissions, moderation | High for community dynamics; OASIS has `JOIN_GROUP`/`CREATE_GROUP` actions that could be leveraged |

For LemonFish's current use cases (e.g., predicting reception of a padel centre in Wollongong), **LinkedIn would probably add the most real signal** — it would capture council officers, club presidents, property developers in their professional voice. Instagram and TikTok are more aspirational given the text-only LLM constraint.

## Part 3 — Extensibility assessment

**Overall: 6/10 extensible.** Not designed for easy addition of new platforms.

### 3.1 What helps

1. Enum-based system — `ActionType` and `RecsysType` are enums, easy to add new values
2. Pluggable recommendation functions — different `rec_sys_*` functions can be swapped
3. Configuration-based platform behaviour (constructor parameters)
4. Unified database schema — all platforms share the same SQL tables (`post`, `comment`, `like`, `dislike`, `follow`, `trace`, `rec`)

### 3.2 What hurts

1. **Hardcoded conditionals everywhere.** Platform logic uses `if self.recsys_type == RecsysType.REDDIT: ... elif ...` scattered across ~50 locations in `platform.py`. Adding a 3rd platform means modifying every action method.
2. **No abstract base class.** `Platform` is concrete, not designed for subclassing.
3. **Tight coupling between recsys_type and social behaviour** — `allow_self_rating`, `show_score`, `refresh_rec_post_count` are platform-specific configs, not isolated into a clean strategy.
4. **Agent generators are separate functions** — `generate_twitter_agent_graph()` and `generate_reddit_agent_graph()` are hand-written twins with no parameterisation.
5. **System message templates are platform-specific** — `UserInfo.to_system_message()` has explicit Twitter vs Reddit conditionals in `config/user.py:48-102`.

### 3.3 Work required to add a new platform (e.g. LinkedIn)

**Estimated effort: 200-300 lines of new code + 50+ modified existing lines** (~2-3 days of focused work).

Required changes:

1. **Add enum value** — `LINKEDIN = "linkedin"` in `typing.py` (1 line)
2. **Implement `rec_sys_linkedin()`** in `recsys.py` — define feed algorithm (50-100 lines)
3. **Add platform init branch** in `env.py:77-98` (10 lines)
4. **Modify ~50 action methods** in `platform.py` to add `elif self.recsys_type == RecsysType.LINKEDIN:` cases (100-150 lines of modifications)
5. **Create `generate_linkedin_agent_graph()`** in `agents_generator.py` (30-50 lines)
6. **Define `get_default_linkedin_actions()`** in `typing.py` (10 lines)
7. **Add LinkedIn system message template** in `config/user.py` (20-30 lines)

**Files to modify:**
```
oasis/social_platform/typing.py         — enums + action subsets
oasis/social_platform/platform.py       — major refactor (~50 method updates)
oasis/social_platform/recsys.py         — new rec_sys function
oasis/environment/env.py                — init branch
oasis/social_agent/agents_generator.py  — new generate_*_agent_graph function
oasis/social_platform/config/user.py    — system message template
```

### 3.4 A cleaner architecture would look like this

If we ever do fork, the right refactor is:

1. Abstract `BasePlatform` class with platform-specific methods as abstract
2. Concrete `TwitterPlatform`, `RedditPlatform` (and future) subclasses
3. Strategy pattern for recsys selection (already partially present)
4. Platform-specific conditionals moved into subclass implementations
5. A platform registry so third parties can plug in without touching core

## Part 4 — Options for LemonFish

Given the engine constraints, we have four realistic options:

### Option A — Accept Twitter + Reddit as the simulation substrate

Document to users clearly: "Twitter simulates algorithmic viral dynamics; Reddit simulates communal voting dynamics. These cover ~80% of what matters for opinion simulation. Your topic plays out on both."

- **Effort:** zero
- **Risk:** low
- **Coverage:** good for opinion dynamics, weak for professional/B2B scenarios
- **Recommendation:** **Yes for v1.0.** Honest, shipping-friendly.

### Option B — Fork camel-oasis and add LinkedIn

- **Effort:** 2-3 days of initial work + ongoing maintenance
- **Risk:** medium — upstream updates break our fork
- **Coverage:** big improvement for business simulations
- **Blocker:** we committed in "Not Doing" that we wouldn't fork. Reversing requires justification.
- **Recommendation:** revisit only if B2B simulations become a core use case.

### Option C — Cross-platform narrative extrapolation in the ReportAgent

Have the ReportAgent produce a "How this would play out on LinkedIn / Instagram / TikTok" section based on the Twitter+Reddit discourse. This is **LLM role-playing LinkedIn** rather than a proper simulation — cheap but epistemically weaker.

- **Effort:** 1-2 hours — add a new ReportAgent section prompt
- **Risk:** low
- **Coverage:** best-effort, clearly flagged as "extrapolation not simulation"
- **Recommendation:** **Yes, as a supplement to Option A.** Low cost, real value.

### Option D — Wait for upstream

camel-ai is actively developed and may add platforms. We could contribute upstream rather than fork.

- **Effort:** zero in the short term, contribution effort later
- **Risk:** depends on upstream priorities
- **Recommendation:** track upstream releases; check every few months.

## Part 5 — Immediate wins status

### 5.1 Wire orphaned fields into the agent prompt — **SUPERSEDED**

Originally this section recommended wiring `profession` and `interested_topics` into OASIS's `other_info` dict. The Phase 7.16 work supersedes this: by appending the personality narrative (which mentions profession implicitly in the persona prose) directly to the `persona` string, we bypass the `other_info` injection path entirely. `profession` and `interested_topics` remain display-only fields, which is fine — the behavioural information they carried is now redundant with the richer `persona` string.

### 5.2 Differentiate institutional personas — **SHIPPED (Phase 7.16)**

The `age=30 + gender=other + mbti=ISTJ` lock on institutions has been replaced with:
- **8 organisational archetypes** (authoritative, technocratic, advocacy, commercial, community, media, academic, populist)
- **5 behavioural dimensions** (formality, risk tolerance, transparency, responsiveness, ideological intensity) scored 0-100

The LLM picks an archetype grounded in the source material and assigns trait scores. The narrative is then appended to the persona string so the agent sees its own behavioural profile in its system prompt. See `docs/personality_frameworks.md` for the full taxonomy.

### 5.3 Persona reproducibility — still open

Add a seed to profile generation (Phase 6 already does this for model assignment — extend to profile generation). This enables reproducible experiments across runs. Low priority until someone actually needs it for published research.

## Summary

- **Personas have 20 fields** (Phase 7.16 added `big_five`, `org_archetype`, `org_traits`) but the behavioural pipeline still routes through the `persona` string alone. We inject the new frameworks by appending a structured narrative paragraph to `persona` during generation.
- **MBTI is now legacy/display only.** Individuals use Big Five / FFM (5 dimensions 0-100, empirically grounded). Institutions use one of 8 archetypes + 5 behavioural trait scores. See [`personality_frameworks.md`](./personality_frameworks.md).
- **Twitter now gets personality info too.** Previously only Reddit's OASIS system prompt mentioned MBTI/age/gender/country. Under the new approach, both platforms receive the full personality narrative via the shared `user_profile` string.
- **OASIS supports Twitter and Reddit only.** No LinkedIn, Instagram, TikTok, Bluesky. Adding platforms is a 2-3 day refactor of upstream code we committed not to fork.
- **For v1.0, accept the two-platform constraint** and document it. Consider cross-platform extrapolation in the ReportAgent as a cheap supplement.
- **Persona reproducibility** remains open — add a seed to profile generation if reproducibility becomes important for publication.
